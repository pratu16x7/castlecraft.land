Approach 1: Just find

Simple Rule 1:

So the bot can scan places, and check if the remote is there.

If found Beep. If not, it will just move on to a new place.

"move to a new place" has a whole bunch of baggage. How will it? Is this the place to discuss it, or as we have done, some way ahead?

make algorithms as graphic as you can, they are pretty hard to parse and navigate.

may be a flow chart (perfect but too technical), but a moving one will help, or one along with a small animation in miniature that shows what it (bot/human) looks like following it

Algo 1:

  1. go to a/next place
  2. search if thing there
  3. if not
    1. go to STEP 1
  4. If Yes
    1. Beep! and STOP

You show this to the bot. and it does it’s thing. After 5 min, it has found the remote. Not exceptional, but it was without any effort on your part, so yeah.

Yeah, the **steps -> visual emergence** will be hard to avoid

whatever story explanantion you have to beget. It's the cleanest approach to understanding the essence of the solution

And to get to the steps, you need some intuitive simple rules, that take the form of:

what is needed/happened/problem -> simple solution/what to do

- eg: can't search all at once -> break down places to search
- eg: search all places / no searching twice -> note down places to search, and mark them off
- eg: no searching twice -> note down places you've been
- eg: if found -> stop the search
- eg: if not found -> go to the next place (in the list)

so in the end, it is **problem -> intuition -> steps -> visual emergence/solution**

But, intuition doesn't happen in one go. It take trial and error, of keeping trying the simple steps we can come up with, until we have a bunch of simple steps that lead to a solution

Ideally, the effect of every simple rule should be shown separately.

Once you notice that it is searching the place it just searched.

Approach 2: The “not-there-here”/“visited” list

Simple Rule 2:

Don’t search the same place twice. Keep a note of the searched (“visited”) places, and if you come across a visited place, don’t search it.

[pic: bot making a list of places it visit, drawing miniature places as list items. Oh graphics will make it easier to transition to a tree later]

Observation 1:

Our list was visited aka “not-there-here” list … with taking snaps after we’d searched the places.

That’s the reason the next time we went there, we didn’t end up duplicating, because we compared what we saw to the list. Oh wait we did take a snap then, to compare. it was whether we added it to the list or not (only if it was not already there.)

Extend previous flow chart.

Algo 2:

  1. go to a/next place
  2. take a snap, and see if it matches any earlier place
  3. if not
    1. search if thing there
    2. if not
      1. add the snap to earlier list
      2. go to STEP 1
    3. if yes
      1. STOP
  4. if yes
    1. go to step 1

Not much time improvement: this doesn’t stop the bot from actually going to that place.

What if: Not just searching again, stop going to a place a second time, Is it possible?

For that, it not only has to search the place just once, it has to go to the place just once.

For that, It should have a organised way to search. Not very obvious, even though 80-90% of the way there

Any way we can look for areas in a way that ensures we go there only once?

Well, that depends on how we choose our next place to search. The next place itself should never be a place that we’ve already visited.

Ooo, this getting terribly close to the singularity :P

And how are we currently doing that?

Randomly.

Eg: I scan, find table, search it. Then maybe I keep moving right, or check the next closest area, or anything like that.

We don’t know if the next such place would be somewhere we’ve already looked.

Trial 2: The Human way

So okay, leave that. Let’s step back a bit.

How’d you, a human, go about it? Well you’d not be perfect, you’d think you’re focusing on searching one area, which is true for a while, but then you’d simply go search the closest place again, rather than taking the effort to move to the next unsearched place. And we don’t even care.

Honestly, looking at the bot runs, the bot too seems to wind up at the same place more often, only ocasionally managing to break free of a closeted area, out of sheer random luck.

[gif: H+B view, with bot mostly searching in hall, and then breaking free and going into bedroom, only to come back

frames: nearly 10-12, with bot positions changing that’s it]

The way you’d most likely resolve this is, if you’re serious enough to properly search the whole place, would be probably (a method I like to call) “-> (break (opt.),) choose, and finishing off -> “. ["break" motivation]

Simple Rule 3: [visual: break]

Can’t search everywhere at once. So before searching, break down a place into smaller parts, and pick one, and finish it off completely before moving on to next.

[gif: breaking an area and zooming in, as a human, 2 passes]

Isn't Break the same as creating artificial/organised opportunity to Choose, which we were going to do in an unorganised way anyway?
